Create `src/services/session/sessionOrchestrator.ts` with these pure functions:

1. `selectStrongestSkill(unlockedSkillIds, skillStates, rng, defaultElo?)` -- Inverts the weakness weighting from selectSkill: highest Elo gets highest weight. Formula: `(skillElo - minElo) + STRENGTH_BASELINE` where STRENGTH_BASELINE=50. Uses cumulative distribution sampling with rng. Used for warmup/cooldown confidence-building.

2. `selectEasiestTemplate(skillId)` -- Returns the template with the lowest baseElo for a given skill. Throws if no templates found. Used for warmup/cooldown.

3. `getSessionPhase(index, config)` -- Returns 'warmup' if index < warmupCount, 'practice' if index < warmup+practice, 'cooldown' if index < total, otherwise 'complete'. Pure derived value from index.

4. `generateSessionQueue(skillStates, config, seed)` -- Generates the full problem queue. Implementation approach per research Pitfall 1 (Elo drift):
   - Pre-generate warmup (3 problems): selectStrongestSkill + selectEasiestTemplate + generateProblem + formatAsMultipleChoice
   - Pre-generate cooldown (3 problems): same as warmup
   - Practice problems (9) are generated LAZILY by the hook (not here) -- return only warmup+cooldown in the pre-generated queue
   - This avoids Elo drift: practice problems use updated Elo after warmup answers

   Actually, per the research recommendation: generate warmup and cooldown upfront (they use easiest templates regardless of Elo), but leave practice slots as null/empty for lazy generation. Return `{ warmup: SessionProblem[], cooldown: SessionProblem[], seed: number }` so the hook can generate practice problems on-the-fly.

   Wait -- this complicates the interface. Simpler approach: generate ALL 15 upfront but accept the minor Elo drift (warmup is 3 easy correct answers, Elo change is ~5-15 points with K=16-40, which barely shifts template selection). The research says: "Accept the minor drift since warmup is only 3 easy problems and Elo change is minimal." Go with full pre-generation for simplicity.

   So: `generateSessionQueue` returns `SessionProblem[]` of length 15. Uses createRng(seed). For warmup (indices 0-2): selectStrongestSkill + selectEasiestTemplate. For practice (indices 3-11): selectSkill (weakness-weighted from adaptive) + selectTemplateForSkill (gaussian-weighted from adaptive). For cooldown (indices 12-14): same as warmup. Calls getUnlockedSkills for the available skill pool. Uses getOrCreateSkillState for Elo lookups.

5. `commitSessionResults(pendingUpdates, totalXp, updateSkillState, addXp)` -- Iterates pending updates Map, calls updateSkillState for each skill with newElo/attempts/correct, then calls addXp(totalXp). Pure side-effect function.

Create `src/hooks/useSession.ts` implementing the session progression hook:

**State:**
- `sessionQueue: SessionProblem[]` stored in useRef (ephemeral, not in Zustand store -- per research anti-pattern guidance)
- `currentIndex: number` in useState
- `feedbackState: { visible: boolean; correct: boolean } | null` in useState
- `pendingUpdates: Map<string, PendingSkillUpdate>` in useRef (accumulated Elo/XP)
- `totalXpEarned: number` in useRef
- `frustrationState: FrustrationState` in useRef (session-scoped per established pattern)
- `feedbackTimerRef: useRef<ReturnType<typeof setTimeout> | null>` for cleanup
- `sessionStartTime: number` in useRef (Date.now() at start)
- `isComplete: boolean` in useState

**Initialization (on mount or when called with startSession):**
- Read skillStates from useAppStore
- Call store's startSession() to set isSessionActive=true
- Generate seed from Date.now()
- Call generateSessionQueue(skillStates, DEFAULT_SESSION_CONFIG, seed)
- Store result in sessionQueue ref
- Initialize frustrationState via createFrustrationState()
- Set currentIndex to 0

**Exposed API:**
- `currentProblem: SessionProblem | null` -- derived from sessionQueue[currentIndex], null when complete
- `currentIndex: number`
- `totalProblems: number` (15)
- `sessionPhase: SessionPhase` -- derived via getSessionPhase(currentIndex, config)
- `feedbackState` -- for UI to show correct/incorrect indicator
- `isComplete: boolean`
- `score: number` -- running count of correct answers
- `handleAnswer(selectedValue: number): void` -- processes answer selection
- `handleQuit(): void` -- discards session, calls store endSession()
- `sessionResult: SessionResult | null` -- populated when complete

**handleAnswer logic:**
1. Get current problem from queue
2. Determine isCorrect = selectedValue === currentProblem.problem.correctAnswer
3. Set feedbackState to { visible: true, correct: isCorrect }
4. Record answer in store via recordAnswer({ problemId, answer, correct, format: 'mc', bugId from selected choice if wrong })
5. Update Elo: call calculateEloUpdate(currentElo, templateBaseElo, isCorrect, attempts) -- store result in pendingUpdates Map (accumulate per skillId)
6. If correct: add calculateXp(templateBaseElo) to totalXpEarned ref
7. Update frustrationState via updateFrustrationState(state, skillId, isCorrect)
8. Set feedback timer (FEEDBACK_DURATION_MS = 1500):
   - On timeout: clear feedbackState, check if currentIndex+1 >= totalProblems
   - If complete: set isComplete=true, call commitSessionResults, call store endSession(), build SessionResult
   - If not complete: increment currentIndex
9. Store timer ref for cleanup

**handleQuit logic:**
- Clear feedback timer if active
- Call store endSession() -- discard everything
- Do NOT call commitSessionResults
- Reset local state

**Cleanup (useEffect return):**
- Clear feedback timer ref (defense-in-depth per CLAUDE.md)

**Important details per user decisions:**
- Elo/XP accumulate in refs during session, commit ONLY on successful completion (not on quit)
- feedbackState drives the 1.5s display before auto-advance
- bugId: when answer is wrong in MC, find the matching ChoiceOption by selectedValue and extract its bugId for misconception tracking
- FrustrationState is managed in ref, checked via shouldTriggerGuard when generating practice problems -- BUT since we pre-generate the queue, frustration doesn't affect problem selection mid-session. This is acceptable for v0.1 since the queue is pre-generated. Note: if frustration guard integration during practice is desired, it would require lazy generation. For v0.1, pre-generated queue is simpler and the frustration guard will be fully utilized in future versions with lazy generation.
