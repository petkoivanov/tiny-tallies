  <action>
    Apply the same undo-wiring pattern to each of the 6 manipulatives. The pattern:

    **For each component:**
    1. Import `useActionHistory` from `../shared`
    2. Replace the primary `useState` call with `useActionHistory`:
       - Before: `const [state, setState] = useState<T>(initial);`
       - After: `const { state, canUndo, pushState, undo, reset } = useActionHistory<T>(initial);`
    3. Replace every `setState(newValue)` call with `pushState(newValue)` -- these are the user actions that should be undoable
    4. Wire `handleUndo` callback that calls `undo()`
    5. Update the `handleReset` to call `reset(initialState)` instead of `setState(initialState)`
    6. Pass `onUndo={handleUndo}` and `canUndo={canUndo}` to `ManipulativeShell`

    **Component-specific details:**

    **Counters** (`CounterState[]`):
    - Replace `const [counters, setCounters] = useState<CounterState[]>([])` with `useActionHistory<CounterState[]>([])`
    - `handleAdd`: `pushState([...counters, newCounter])`
    - `handleFlip`: `pushState(counters.map(c => ...))`
    - `handleMove`: `pushState(counters.map(c => ...))`
    - `handleReset`: `reset([]); nextId.current = 0;`
    - `handleUndo`: `const prev = undo(); if (prev) /* haptic */`

    **TenFrame** (`boolean[]`):
    - Replace `const [cells, setCells] = useState<boolean[]>(...)` with `useActionHistory<boolean[]>(...)`
    - `handleSnap`: `pushState(next)` where next is the new cells array
    - `handleCellTap`: `pushState(next)`
    - `handleReset`: `reset(new Array(CELLS_PER_FRAME).fill(false)); setFrameCount(1); ...`
    - Keep `frameCount` as separate useState -- it auto-derives from cells, not independently undoable
    - `handleUndo`: call `undo()`, then recalculate frameCount from restored cells

    **BaseTenBlocks** (`BlockState[]`):
    - Replace `const [blocks, setBlocks] = useState<BlockState[]>([])` with `useActionHistory<BlockState[]>([])`
    - CRITICAL: In `handleUndo`, call `clearTimer()` BEFORE `undo()` to prevent auto-group race condition (per RESEARCH.md Pitfall 1)
    - `onGroupEnd` in useAutoGroup: the auto-group callback should use `pushState` so auto-group itself is an undoable action (undo restores 10 cubes from the rod)
    - `handleReset`: call `clearTimer()` then `reset([])`
    - Keep `blocksRef.current` in sync: after `pushState`, update `blocksRef.current = newBlocks`

    **NumberLine** (marker position state):
    - Replace marker position state with `useActionHistory`
    - Push state on marker drag end (onEnd event)
    - Undo restores previous marker position

    **FractionStrips** (strip shading state):
    - Replace strip sections state with `useActionHistory`
    - Push state on tap-to-shade/unshade
    - Undo restores previous shading pattern

    **BarModel** (bar sections state):
    - Replace bar sections/divider state with `useActionHistory`
    - Push state on divider drag end and label changes
    - Undo restores previous bar configuration

    **Verification pattern for each component:**
    After wiring, visually verify by running typecheck. Each component should:
    - Accept undo via ManipulativeShell (undo button visible in header)
    - Not break any existing behavior (add, drag, snap, reset still work)

    Keep all files under 500 lines. If any file approaches 500 lines after changes, extract helpers.
